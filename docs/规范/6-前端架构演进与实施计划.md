# 🚀 前端架构演进与纯前端实施计划

> **更新说明**：经过评估，由于目前项目主要为个人展示与静态工具集合，引入 Supabase/数据库 会大幅增加运维成本与复杂度。当前阶段**不需要数据库**，所有功能均采用“纯前端 + 本地/静态化”方案实现。

## 🎯 为什么现在不需要数据库？
1. **内容静态化优势**：知识库与工具目前完全可以通过静态 JSON (`features.ts`) 和 Markdown/HTML 文件来维护，配合 Next.js 的服务端渲染，拥有极高的访问速度和极低的运维成本。
2. **状态本地化**：像时间管理、人生轨迹的进度等，如果只是个人查看，完全可以硬编码或存储在 `localStorage` 中。
3. **安全与轻量**：没有数据库就不需要考虑接口鉴权、防注入、存储被刷等攻击，专心打磨 UI/UX。

## 🛠️ 纯前端阶段实施计划

### Phase 1: 悬浮工具箱与全局导航 (当前重心)
将原计划中的复杂工具箱简化为前端组件：
- **目标**：在右下角提供一个常驻的悬浮动作按钮 (FAB)。
- **核心功能**：
  - 点击后展开 `cmdk` 驱动的全局快捷面板。
  - 支持快捷切换路由（首页、众妙、关于）。
  - 支持快捷切换系统主题（日/夜）。
- **执行**：安装 `cmdk` 与 `framer-motion`，封装 `GlobalToolkit.tsx` 并挂载至根节点。

### Phase 2: 轨迹扩展与时间规划
- **目标**：在现在的“人生轨迹图 (`JourneyOverlay.tsx`)”中补充规划视图。
- **实施**：用前端静态数组维护“未来规划”节点。利用前端库（如 FullCalendar 或原生手写）在 Journey 组件的 Tabs 里加一栏“未来规划”，实现时空的对接。

### Phase 3: 单页 HTML 沉淀与知识库打通
- **目标**：统一“知识库”的入口体验。
- **实施**：将 `docs/单页HTML/` 下的文件作为静态资源放置到 `public` 目录下，在 `FeatureWindow` 里点击对应文件时，使用 iframe 直接无缝嵌入展示。

*注：后续若产生如“评论留言”、“文章自动发布 CMS”等强后端需求，再考虑引入 Supabase，目前专注打造极致的静态前台体验。*
