# 其他必要规范

除了目录、API和命名规范外，项目在开发过程中还需要遵守以下通用规范以保证代码质量。

## 1. 代码格式化与 Linting

- 项目已配置 `ESLint` 和 `TypeScript`。
- **必须**在提交代码前解决所有的 Lint 错误和类型警告。
- 建议在编辑器（如 VS Code）中开启保存时自动格式化（Format On Save），以保证团队代码风格统一。

## 2. TypeScript 使用规范

- **避免使用 `any`**：尽量为所有变量、函数参数和返回值定义明确的类型。确实无法确定类型时，可优先考虑使用 `unknown`。
- **接口与类型别名**：优先使用 `interface` 来定义对象的形状，对于联合类型或复杂交叉类型使用 `type`。
- **公共导出**：为独立或公用的 Type/Interface 创建单独的 `.d.ts` 文件，或放置在统一类型目录下集中管理。

## 3. Git 提交流程规范

- 提交信息 (Commit Message) 必须清晰描述本次提交的目的。建议遵循 Angular 提交规范。
- **格式**：`<type>(<scope>): <subject>`
  - `feat`: 新增功能
  - `fix`: 修复 bug
  - `docs`: 文档变更
  - `style`: 代码格式调整（不影响代码逻辑的空白、格式化、缺失分号等）
  - `refactor`: 代码重构（既不修复 bug 也不添加新功能的代码更改）
  - `perf`: 优化相关（提升性能、体验等）
  - `test`: 增加测试或更新测试
  - `chore`: 构建过程或辅助工具的变动

## 4. 注释规范

- **必要性**：代码应尽量做到自解释，但对于复杂的业务逻辑、算法实现或 hack 写法，必须加上清晰的注释。
- **函数注释**：公共函数或复杂函数建议使用 JSDoc 格式进行注释，说明功能、参数及返回值。
- **TODO 注释**：遇到暂未实现或需要优化的代码，使用 `// TODO: [描述] [操作人] [日期]` 格式标记，并定期清理。

## 5. 组件开发与代码整洁之道 (Clean Code)

- **单一职责原则 (SRP)**：每个组件/函数应尽量只负责一件事情。如果组件过于臃肿（超过 300 行）或函数过长（超过 20 行），应考虑拆分。
- **避免硬编码与魔术数字**：尽量将文案、配置项、魔法数字提取为常量或统一配置。
- **KISS & YAGNI**：保持简单，不为未来“可能”用到的功能过度工程化。
- **扁平化结构**：优先使用防御性编程（Guard Clauses，提前 Return），避免过深的嵌套（尽量不超过 2 层）。

## 6. Next.js 与 React 性能优化

- **消除瀑布流请求**：对于不相互依赖的数据获取，使用 `Promise.all()` 并行请求。
- **控制 Bundle Size**：对于体积较大的第三方库或非首屏关键组件，优先使用 `next/dynamic` 进行动态导入，避免使用导致全量打包的桶导出（Barrel Imports）。
- **服务端优先**：默认使用 Server Components (RSC)，仅在需要交互时才标记 `"use client"`。
- **缓存组件 (Next.js 16+)**：对于频繁访问且不常变动的数据，利用 `"use cache"` 缓存指令优化响应速度。
- **原生表单 (`next/form`)**：在 Next.js 16+ 中，对于搜索、过滤等 GET 请求，一律优先使用 `next/form` 而非原生 `<form>` 以获得渐进式增强及 URL 自动同步体验。
