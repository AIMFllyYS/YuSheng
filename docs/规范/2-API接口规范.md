# API接口规范

为确保前后端数据交互的稳定性与一致性，本项目的 API 接口需要遵循以下规范：

## 1. 基础规范

- **通信协议**：统一使用 `HTTP/HTTPS` 协议。
- **数据格式**：请求和响应的数据格式统一为 `JSON` (即 `Content-Type: application/json`)。
- **字符编码**：统一采用 `UTF-8` 编码。

## 2. URL 设计规范 (RESTful API)

- 接口地址应使用名词，并采用小写字母，多单词之间使用中划线 `-` 分隔，例如 `/api/user-profiles`。
- 层级关系表达清晰，例如获取某个用户的文章集合：`/api/users/{userId}/articles`。

## 3. HTTP 请求方法

- `GET`：用于从服务器获取资源。
- `POST`：用于向服务器新建资源。
- `PUT`：用于更新服务器上的完整资源。
- `PATCH`：用于更新服务器上的部分资源。
- `DELETE`：用于删除服务器上的资源。

## 4. 响应数据结构规范

无论请求成功与否，后端返回的 JSON 数据结构应保持一致，包含以下标准字段：

```json
{
  "code": 200,             // 状态码：200 表示成功，其他表示各类业务错误或系统错误
  "message": "请求成功",   // 提示信息：用于前端显示或调试
  "data": {                // 实际负载数据：可以是对象、数组或 null
    "id": 1,
    "name": "AIMFlly"
  }
}
```

## 5. 错误处理与状态码定义

- `200`: 请求成功。
- `400`: 请求参数错误 (Bad Request)。
- `401`: 未授权，需要用户登录 (Unauthorized)。
- `403`: 拒绝访问，权限不足 (Forbidden)。
- `404`: 请求的资源不存在 (Not Found)。
- `500`: 服务器内部错误 (Internal Server Error)。

前端需封装全局的请求拦截器与响应拦截器，集中处理 HTTP 状态码与业务 `code` 的异常提示。
